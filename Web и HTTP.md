## Что происходит при вводе адреса в браузере?
- поиск в кэше браузера
- поиск файла в локальной файловой системе, если начинается на file://
- gethostbyname (сначала поиск в файле /etc/hosts)
- gethostbyname (затем обращение к кэшу провайдера)
- корневой dns (например, site.com.ru - dns ru сначала), процесс поиска через корневой dns называется **dns lookup**
- установка tcp-соединения посредством рукопожатия (syn, ack, syn-ack)
- загрузка html-кода страницы
- затем браузер парсит этот код и если встречает ресурс, то сразу начинает пытаться его загрузить

## В чем отличие QUIC (HTTP3)?
- быстрая установка соединения (TCP требует рукопожатия, а это время (RTT-round-trip-time) при удаленности клиента и сервера >100мс)
- меньше блокировок HoL
- QUIC использует использует протокол UDP, потому что это позволяет не обновлять огромное количество устройств, которые уже знакомы с UDP, то есть так QUIC было проще развернуть

## Что такое NAT?
- **Network Address Translation** - это механизм в сетях TCP/IP, позволяющий изменять IP адрес в заголовке пакета, проходящего через устройство маршрутизации трафика.
- **NAT** переводит приватные адреса, в общедоступные. Это позволяет устройству с частным адресом IPv4 обращаться к ресурсам за пределами его частной сети.
- **как NAT обеспечивает безопасность?**
	- только если порты открыты и не совсем обеспечивает

## Pooling vs Long Pooling vs Server Site Events vs WebSockets?
- **pooling или частые запросы** - это просто каждые N секунд клиент шлет GET запрос на сервер, чтобы понять, например, есть ли новые сообщения.
- **long pooling или долгие запросы** - это когда клиент отправляет запрос и сервер не сразу закрывает соединение, а ждет некоторый таймаут и если сообщение появляется раньше, то сразу его отправляет и клиент немедленно отправляет новый запрос.

## NGINX
##### это **обратный прокси сервер**, который выполняет следующие задачи:

- **балансировка** запросов между несколькими внутренными сервисами (иногда это делает балансировщик)
- **безопасность** - чтобы предотвратить доступ к ресурсу из интернета, защитить от атак
- **шифрование** TLS (transport layer security over IP), SSL (secure socket layer over IP)
- **кэширование** контента веб страниц или определенных ресурсов для снижения нагрузки на веб сервисы

> проще говоря, он позволяет отдавать статические ресурсы сервера (картинки, файлы и тд), в более общем (сложном) случае он позволяет еще и выполнять сжатие, кэширование, балансировку и тд
##### **как он работает?**
- **главная идея** в том, что мы имеем **мастер процесс и набор воркеров**, которые по дефолту установлены по числу процессорных ядер (worker_processes auto) и эти воркеры обрабатывают одновременно до 1024 соединений
- при инициализации каждый рабочий процесс начинает работать с кучей прослушивающих сокетов, унаследованных от главного процесса - главный процесс работает как гроссмейстер, который играет много шахматных партий одновременно, то есть так же, как оператор select в go при чтении из каналов
