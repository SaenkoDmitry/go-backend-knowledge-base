## Что мониторить?
### Pod metrics
**ключевые метрики здоровья пода должны писаться независимо в виде подключаемой библиотеки:**
- **CPU** - количество потребляемой памяти по отношению к общей памяти пода, подумать, когда происходят пиковые нагрузки в течении дня и предусмотреть их
- **CPU throttling** - тротлинг процесса - снижение перфоманса на пиковой нагрузке из-за перегрева процессора
- **memory** - количество потребляемой памяти
- **threads** - количество созданных потоков
- **fault / errors rate (page/widget/http/grpc)** - общее количество ошибок по странице или виджету или http или grpc ручке
- **incoming TCP connections rate** - количество открытых входящих соединений
- **outgoing TCP connections rate** - количество открытых исходящих соединений
- **network bandswitch** - (mbps) - количество данных, переданных по сети в заданный промежуток времени
- **total established TCP connections** - общее количество открытых соединений в данный момент времени
- **TW connections rate** - (time-wait) количество ожидающих выполнение соединений (здесь подумать об особенностях сервера в go
- **RT response time** - (latency by percentiles) - среднее время ответа на 50, 90, 95, 99 персентилях
- **RPS** - (request per second, total / by pods) - количество обрабатываемых запросов в секунду
### KV-storage metrics
- **hits / misses cache - (redis or other KV-storage)** - количество попаданий / промахов в кэше, чтобы понимать, насколько он эффективен и не возникли ли какие-либо проблемы с кэшированием
### DB metrics
- **max tx duration** - максимальное время транзакций в определенный момент времени
- **transactions** - количество транзакций на кластер / конкретную базу данных
- **pgbouncer active connections** - количество активных соединений балансировщика
- **pgbouncer idle connections** -
- **pgbouncer client waits connections** -
- **pgbouncer query count** -
- **cpu usage** - потребление cpu
- **memory usage** - потребление памяти
- **database size** - размер базы данных по отношению к доступному месту на диске
- **disk writes** - количество записей на диск в определенный момент времени
- **disk reads** - количество чтений с диска в определенный момент времени
- **rows fetched** - количество извлеченных строк
- **rows returned / fetched** - количество извлеченных строк по отношению к тому, которое было возвращено ( может помочь при оптимизации запросов, чтобы оптимизировать перфоманс )
- **pg_stat_replication_pg_wal_lsn_diff** - что-то про лаг репликации, когда обновление мастера не сразу становится доступнуым на репликах
- **locks** - (sharelock, rowsharelock, exclusivelock, accesssharelock, rowexclusivelock, accessexclusivelock, sharerowexclusivelock, shareupdateexclusivelock) - разного рода блокировки

## Как мониторить?

### стандарт де-факто:
**prometheus** ( time-series субд для хранения логов ) + **grafana** ( инструмент для визуализации метрик, в том числе позволяющий смотреть логи с машин )
### порядок настройки такой:
- поднять prometheus, который будет хранить метрики приложений
    - он сам тянет метрики из заданного множества сервисов, за  счет чего у него не может забиться очередь и метрики никогда не станут узким местом производительности системы
    - не предполагает механизма горизонтального масштабирования или high availability, если не будет справляться, то надо просто взять машину помощнее
    - подробнее здесь - [https://eax.me/prometheus-and-grafana/](https://eax.me/prometheus-and-grafana/)
- поднять node-exporter для того, что хотим мониторить в виде, понятном prometheus:
    - например, можно установить на линукс машину node-exporter, чтобы он поднял на машине страничку с метриками по локейшну /metrics
    - либо же добавляем в go зависимость, которая также создаст нужный хэндлер и в который можно будет внутри приложения добавлять кастомные метрики, подробнее посмотреть можно здесь - [https://prometheus.io/docs/guides/go-application/](https://prometheus.io/docs/guides/go-application/)
- поднять grafana, который представляет собой веб-фронтенд к time-series субд, вроде prometheus, graphite, influxdb
