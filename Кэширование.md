## **Стратегии работы с кэшом**

### **cache aside (ленивая загрузка)**
> **READ PATH**: Приложение само управляет чтением из cache и загрузкой из DB при miss.
> Если промах → читаем из БД → синхронно обновляем кэш.
> **WRITE PATH:** Запись сначала в DB, затем invalidate/update cache для контроля stale-данных.
> **⚠ Критично** не забывать invalidation, иначе растет stale-read rate.
###### **thundering herd problem / cache stampede**
> ситуация, когда у горячего ключа одновременно истекает TTL и множество запросов идут в DB.
> **Решения**: single flight, jitter, refresh-ahead, distributed lock.

### **read through (сквозное чтение)**
> **READ PATH**: Приложение просто делает get, а вся логика чтения **инкапсулирована в отдельном слое** (proxy/middleware, библиотека).
> **⚠ Риск**: Сложность переносится в cache middleware / собственный proxy слой и требует хорошей observability.

### **write through (сквозная запись)**
> Каждая запись синхронно обновляет и cache, и DB, обеспечивая предсказуемые чтения (read-after-write consistency).
> cache и DB могут расходиться при сбоях, поэтому нужна компенсационная логика.
> Обычно пишут **сначала в базу, потом в кэш**. Если база упала — кэш не обновляем.
> **⚠ Риск**: Write latency растет, и важно корректно обрабатывать **partial failures**.

**READ PATH**: Чтения обычно идут из cache и имеют стабильную низкую latency.
**WRITE PATH**: Write подтверждается только после успешной записи в cache и DB.
###### **А что если кэш записался, а DB упала или наоборот (partial failure)?**
- придется совершить усилия по откату изменений:
	1) **cache→db: Кэш записался, DB упала** → синхронно invalidate кэш, клиент получает 500.
	2) **db→cache: DB записалась, кэш не записался** → чтение пойдёт в DB (cache miss), данные не потерялись

### **write behind (write back, асинхронная запись)**
> Приложение пишет в кэш сразу, а затем асинхронно батчами обновляет БД.
> **⚠ Риск потери последних записей** если нет надежного буфера (так как копим где-то для батчирования записи).

**READ PATH:** сразу из кэша.
**WRITE PATH**: Клиент получает быстрый ack после записи в cache, без ожидания DB.
Отложенный flush в DB позволяет сглаживать write spikes и делать batching.
Нужны durable buffer и recovery-процедуры при сбоях.

> В production write-behind обычно реализуется через **persistent message queue**, но может быть встроенный механизм cache (redis AOF).

### **refresh ahead (опережающее обновление)**
> Кэш периодически **обновляется до истечения TTL**, чтобы данные всегда были “горячими”.
> background worker таргетировано обновляет ключ, но только если он активно читается.
###### **Когда применять**
если много чтений, а записей мало, можем обновлять данные в кэше хитрым образом: то есть не всегда получать обновления от базы, а только если прошло какое-то время (например, 0.5 TTL ИЛИ probabilistic refresh (как в CDN)) и ключ все еще горячий — только в этом случае идем и обновляем значение.
###### **Как определить 'горячий ключ'**
Счетчики обращений, sliding window.
 в Go для трекинга горячих ключей, как правило, используется sliding window через `time.Ticker` + `sync.Map` с atomic counters.

### **Write-Around**
> Запись идет только в DB, кэш не обновляется.
###### **Когда применять**
write-heavy workloads, где большинство записанных данных никогда не читаются (например, логи, event stream).

### **Короткое правило выбора**
- если важнее предсказуемость и простота - начинайте с **Cache-Aside**;
- если критичен строгий read-after-write - смотрите в сторону **Write-Through**;
- если критичен write throughput (высокая скорость записи) и допустим eventual consistency - используйте **Write-Behind** с надежным flush-контуром.

### **Консистентность каждой стратегии**
| Стратегия     | Консистентность                                                 | Потеря данных       |
| ------------- | --------------------------------------------------------------- | ------------------- |
| Cache-Aside   | Eventual                                                        | Нет                 |
| Read-Through  | Eventual                                                        | Нет                 |
| Write-Through | Read-after-write (в рамках cache, не гарантирует global strong) | Нет                 |
| Write-Behind  | Eventual                                                        | Возможна (до flush) |
| Refresh-Ahead | Eventual                                                        | Нет                 |
| Write-Around  | Eventual                                                        | Нет                 |
### **Эффективность хранилищ данных**

- CDN ~5ms
- Redis/Memcached ~ 1-5ms
- Local Cache ~0.1ms
- Database ~10-100ms и выше

### **Go инструменты**

- golang.org/x/sync/singleflight – для предотвращения thundering herd problem
- [ristretto](https://github.com/dgraph-io/ristretto) / [bigcache](https://github.com/allegro/bigcache) — in-memory кэши с TTL и eviction policies

## **Уровни кэшей**

если есть некий сайт и человек обращается к приложению через него, то кэши могут быть такие последовательно:
- кэш браузера
- кэш прокси-сервера
- кэш на входе в сайт
- кэш движка сайта
- кэш приложения
- кэш движка БД
- кэш ОС/дисковой системы, где лежит БД

---
## **Стратегии инвалидации кэша**

- TTL
- по событию (например, обновлению в источнике)
- event-driven invalidation (через Kafka)
- versioned keys (user:123:v2)
- delete-on-write (инвалидация вместо обновления)

> 👉 invalidate часто проще и безопаснее, чем обновление

---
## **Стратегии вытеснения кэша**

- **TTL** - удаляется та запись, которой осталось меньше всего жить
- **LRU** - удаляется запись со временем использования старше остальных
- **LFU** - удаляется запись, которая наиболее редко запрашивалась

---
## **Кэширование ошибок / negative caching**

> кэширование "пустых" или ошибочных ответов (например, 404)

#### **зачем:**  
- защита от cache penetration (несуществующие ключи)  
- снижение нагрузки на DB  
  
#### **⚠ важно:**  
- короткий TTL  
- не кэшировать transient ошибки (например, 500)

## **multi-level cache**

> Каждый дополнительный уровень кэша увеличивает риск stale данных.

```
L1: local (in-process)  
L2: Redis  
L3: DB
```
#### **Проблемы**
- invalidation между уровнями
- stale data amplification
#### **Как быть?**
- делать invalidate через событие (pub/sub)  
- использовать versioned keys (например, user:123:v2)  
- уменьшать TTL на L1 (local cache)  
- иногда сознательно допускать eventual consistency

> **stale data amplification** — это ситуация, когда устаревшие данные “размножаются” и живут дольше/шире, чем должны, из-за нескольких уровней кэша.
> **Пример:**
> L1 (local cache, TTL = 1 мин)  
> L2 (Redis, TTL = 10 мин)  
> DB
> **Сценарий:**
> 1. Данные изменились в DB
> 2. Ты инвалидируешь L2 (Redis)
> 3. **НО L1 не инвалидирован**
> **👉 В итоге:**
> - L1 продолжает отдавать stale данные
> - новые запросы даже не доходят до L2
> - stale живёт **дольше, чем TTL L2**

## **serialization cost**

> Иногда cache медленнее из-за:
> - JSON / protobuf encode/decode
> - network overhead
> Иногда выгоднее кэшировать уже сериализованные данные.

## **Hot keys**

> ключи с очень высоким RPS могут перегружать один shard cache  
  
**решения:**  
- replication  
- key sharding (user:123:1, user:123:2)  
- local cache (L1)
