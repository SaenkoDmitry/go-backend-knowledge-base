## Scheduler
> [Планировщик Go — самый подробный гайд простым языком](https://habr.com/ru/articles/891426/)
> [vk video. внутреннее устройство планировщика go](https://vkvideo.ru/video-226668821_456239021)

### как отвечать
>В Go используется **M:N планировщик (GMP модель)**, где goroutines (G) мультиплексируются на системные потоки (M) через логические процессоры (P).
>У каждого P есть локальная очередь задач, плюс есть глобальная очередь. Это уменьшает contention.
>Планировщик использует **work-stealing**: если у P нет задач, он ворует их у других.

> **Goroutine** — это легковесный поток, управляемый рантаймом Go.
> Они **дешевле OS threads**, потому что:
> - маленький стек (растёт динамически)
> - user-space scheduler (not kernel), поэтому context switching дешевле
> Можно создавать десятки тысяч goroutines (порядка 10 000 потоков можно создать на go-программу от ОС)

### Сущности
>- **GRQ** (global run queue) + **LRQ** (local на каждом процессоре) (**зачем разделение?** – в глобальной очереди будут блокировки и хорошо бы их минимизировать)
>- sysmon
>- netpoller
>- thread pool

### Механизмы
> - work stealing (украсть половину задач у соседнего процессора)
> - handoff
> - syscall (системный вызов, который блокирует исполнение потока)

### Недостатки
>- нет приоритетов и нельзя настроить *порядок (по умолчанию горутин порядок FIFO)*;
>- *отсутствие гарантий времени выполнения:* если какая-то горутина будет вычислять что-то сложное вычислять и не будет происходить никаких сигналов к тому, что нужно передать управление другой горутине, то вышеозвученная горутина будет выполняться очень долго, что может привести к *утечке горутин*;
>- горутины могут перемещается между тредами, *снижая эффективность кэшей*.

### Вытесняющий или кооперативный
- С версии Go 1.14 планировщик стал **частично вытесняющим** — long-running goroutines могут быть прерваны.
- **после 1.14 правильно считать вытесняющим**, потому что у него появились такие возможности
	1) как принудительное прерывание горутины,
	2) использование таймеров и других механизмов для определение момента переключения;
	3) **ранее он был кооперативным**, так как горутины сами определяли, когда передать управление другой, например, при работе с каналами
- разница в том, что при **вытесняющей многозадачности** (как, например, на уровне ОС) у нас есть некий регулятор, который может прийти и забрать ресурсы у потока/горутины (это делается через сигналы ОС), а при **кооперативной многозадачности** задачи самостоятельно отдают управление только при определенных событиях (ввод-вывод, блокировка на мьютексе, явный вызов runtime.Gosched)

### GMP-модель
- где G-горутина, M-machine или некая обертка над системным потоком, P-процессор
-  кол-во локальных процессоров P управляется с помощью GoMaxProcs, по дефолту по кол-ву ядер ОС
- **как часто ходить в глобальную очередь?** – **один раз в 61 такт**. почему столько? потому что 61 - **число простое**, не слишком большое, не слишком маленькое. простое,  поэтому меньше коллизий между горутинами.

### жизненный цикл горутины?
- в определенный момент времени процессор P берет на выполнение на потоке M следующую горутину из очереди
- если это **cpu-bound** нагрузка, то задача выполняется
- если же происходит **IO-вызов**, то поток M должен заблокироваться ОС:
	- для этого процессор P может либо попробовать дождаться разблокировки потока, если это короткоживущая операция
	- либо же если поток заблокирован достаточно долго, то процессор берет и открепляет поток от себя и берет следующую горутину и исполняет ее на другом потоке ОС M2 - этот поток берется из отдельного **thread pool-а**
	- за необходимостью открепить поток следит **sysmon**
- **какие состояния бывают у горутины?**
	- running, runnable, waiting
### алгоритм поиска работы процессором?
- 1/61 GRQ
- LRQ
- work stealing
- GRQ
- ... network poller
### что за механизм handoff?
- горутина блокируется на системный вызов, соотв поток тоже блокируется и чтобы не блочить всю локальную очередь горутин процессора, берется новый системный поток и навешивается на очередь горутин
- если считаем, что операция выполнится быстро, то handoff-а не будет
- если превышает лимит, то мы все равно решаем совершить handoff
- кто следит за временем после истечения которого нужно решать, делать или нет handoff? - **sysmon**

### netpoller
- **как сделать так, чтобы системные вызовы не блокировали наши трэды / потоки?**
	- можно воспользоваться системными механизмами для выполнения асинхронных вызовов: **epoll** - linux, **kqueue** - macos
- **как работает?**
	- трэд когда хочет заблокироваться, он не делает это сам, а просто регистрирует это в спец системе (**netpoller**), которая уже сама следит и управляет syscall-ом (горутина при этом переходит в состояние waiting и уезжает в netpoller, а другие продолжают выполнение на процессоре)
- **всегда ли работает?**
	- если работаем с большим файлом, то его нельзя передать epoll-у
- **как горутинам выбраться из netpoller-а?**
	- если они завершают syscall, то они переходят в состояние runnable и тогда их может забрать процессор, а если он этого не сделает, то спустя некоторое время sysmon перенесет их в GRQ
- **при работе с файлами каждый syscall порождает handoff (epoll не спасет) !**

### Syscall в  1.26
- убрали psyscall вообще
- P сразу доступен для других G
- раньше Sysmon раз в 10ms проверяет Psyscall, теперь проверок нет, P сразу откерепляется от M без ожидания когда придет sysmon
##### Для сетевых вызовов
Ничего не изменилось. Netpoller работает как раньше — через `gopark`/`goready`, **без syscall состояния**.
##### Для файловых syscall
```plain
Было:  Psyscall → ждём sysmon → отцепление → overhead
Стало:  P сразу свободен → нет задержки → меньше contention
```
##### старая модель
```plain
G → syscall (read file) → M уходит в ядро
                              ↓
                         P.state = Psyscall
                              ↓
                         [ждём...]
                              ↓
                         sysmon (раз в 10ms)
                              ↓
                         M в syscall > 20μs?
                              ↓ да
                         отцепляем P от M
                              ↓
                         M без P в ядре
                              ↓
                         после syscall: M ищет/создаёт P
```
##### новая модель
```plain
G → syscall (read file) → M уходит в ядро
                              ↓
                         P СРАЗУ освобождается (atomic handoff)
                              ↓
                         другой M берёт P и продолжает работу
                              ↓
                         после syscall: M возвращается
                              ↓
                         G ставится в runqueue (global или другого P)
```

### порядок выполнения горутин
- если горутина выполняется дольше 10мс, то sysmon просит горутину освободить процессор и дать выполнение другой
- **как это работает?**
	- **sysmon** проставляет значение переменной **stackguard**, которую горутина затем видит и понимает, что выполняется уже долго и ей пора освободить выполнение
	- **когда горутина проверяет эту переменную?**
		- вызов любой функции
		- блокирующие операции
- **что будет с горутиной с бесконечным циклом внутри?**
	- до версии 1.14 она зависнет и никогда не завершится
	- после версии 1.14 она будет снята с ядра
- **как завершать горутины?**
	- с помощью сигналов. в go используется SIGURG

## Garbage Collector

> [habr. Go's Garbage Collection: как работает и почему это важно знать](https://habr.com/ru/companies/avito/articles/753244/)

### как отвечать?
> Автоматическое управление памятью.
> В Go используется concurrent mark-and-sweep GC.
> **Основные идеи:**
> - tri-color marking (white/grey/black)
> - write barrier для корректности при мутациях
> - GC работает конкурентно, но есть короткие STW фазы.

### Важно:
> - GC влияет на latency
> - много аллокаций → больше GC → хуже p99

### причины появления GC / что дает?
- сложности ручного выделения и освобождения памяти
- фрагментация памяти
- утечки памяти
- висячие указатели (объекты не используютс  я и не будут, но занимают место)

### trade-off
- потребляет много ресурсов / cpu
- нельзя предсказать момент начала

### стратегии сборки мусора
- подсчет ссылок / reference count - чиселка у каждого объекта, когда 0, тогда удаляем (проблема: циклические зависимости)
- трассирование / tracing - более частая реализация; иде я в том, чтобы искать живые и мертвые объекты по графу

### Версии сборщика мусора в GO
**Go 1.0 (STW mark&sweep, простая реализация)**
- STW mark&sweep
- пробегает по всей куче, удаляет мусор
- паузы > 100 мс; не подходит для low-latency
**Go 1.5 (concurrent mark-sweep, первый конкурентный)**
- GC работает параллельно с программой
- write barriers для безопасной работы с памятью (чтобы GC не терял ссылки)
- паузы ~1-10 мс; пригоден для серверов и API
**Go 1.9 (concurrent with tuning, оптимизации аллокатора)**
- Precise GC: точное отслеживание указателей
- очистка в фоновом режиме без остановки программы (???)
- адаптивность под нагрузку и размер кучи
- паузы строго ~ 1-5 мс
**Go 1.20 (full concurrent GC, почти незаметен, low-latency)**
- GOMEMLIMIT - ручное управление лимитом памяти
- автоматическая оптимизация под нагрузку
- паузы < 0.5-1 мс даже на больших кучах; подходит для стриминга, гейминга, real-time систем
**Go 1.26**
- **Green Tea GC**
- **основная идея**: раньше были спаны для аллокатора (для выделения новых объектов), теперь спаны используются GC для более память-центричного подхода для обхода кучи (то есть сканируем спаны 8 Кб целиком, не бегая по ссылкам)
- **раньше:**
	- проблема старого сборщика в том, что приходилось бегать по разбросанным в разных частях памяти данным
	- процессор постоянно ждал пока данные подгрузятся из оперативной памяти
- **теперь:**
	- данные разбиты на блоки (spans) по 8 Кб, объекты одинакового размера хранятся в одном таком блоке
	- сканируем все объекты в спане ПОДРЯД, если необходимо
	- память-центричный подход вместо процессор-центричного
- то есть, 
	- раньше **единица работы** – объект, затем идем по его ссылкам и прыгаем по памяти, теперь – единица спан (8 Кб) целиком
	- раньше **порядок обхода** – следуем по указателям, теперь линейно по памяти
	- лучше cache locality + преимущество в виде векторизации используется
- на 10-40% быстрее (+ использует преимущества векторизации)

## Arrays & Slices

> - [vk video. все о массивах и слайсах](https://vkvideo.ru/video-226668821_456239027)
> - [habr. Готовим слайсы в Go: подробно о динамических массивах, строчках и ускорении](https://habr.com/ru/companies/yadro/articles/874480/)

### как отвечать
> **Slice** — это структура:
> - pointer на массив
> - length
> - capacity
> - При append может произойти realloc.

> **Почему возможны memory leaks?** потому что slice держит ссылку на большой массив

> В Go **размер среза будет удваиваться** до тех пор, пока он не станет содержать **1024** элемента, после чего будет увеличиваться на **25 %**.

```go
type SliceHeader struct {
		Data uintptr
		Len int
		Cap int
}
// slice содержит указатель на ячейку памяти именно, а не на массив**
```
```go
type StringHeader struct {
		Data uintptr
		Len int
}
// строки иммутабельны, поэтому у SliceHeader нет capacity**
```
```go
stats := new(runtime.MemStats)

bytes := make([]byte, math.MaxInt32)
runtime.ReadMemStats(stats) // 2147596504

// при создании строки происходит дополнительная аллокация памяти
bytesAsString := string(bytes)
runtime.ReadMemStats(stats) // 4295082976

// при передачи в функцию мы аллоцируем место только по StringHeader - 16 байт
```

## Interfaces

> - [habr. погружение в интерфейсы go](https://habr.com/ru/articles/856272/)

### как отвечать
> **Интерфейс** — это (type + data).
> **Под капотом:**
> - указатель на тип
> - указатель на данные
> **nil interface != nil pointer**

```go
type iface struct {
    tab  *itab
    data unsafe.Pointer
}
// itab - указатель на таблицу, содержащую тип и 
// набор указателей на методы типа, удовлетворяющих интерфейсу
// data - указатель на сами данные структуры
```

> **не используйте** интерфейсы для инкапсуляции
> **используйте** интерфейсы для полиморфизма

> **ad-hoc polymorphism** it is not true polymorphism **- true** polymorphism based **on generics**
##### ad-hoc polymorphism

```go
type StringTransformer interface {
    Transform(in string) (out string)
}

type PrefixAdder struct {
    prefix string
}

type PostfixAdder struct {
    postfix string
}

func (s PrefixAdder) Transform(in string) { ... }
func (s PostfixAdder) Transform(in string) { ... }
```

##### для чего нужны?
- уменьшение дублирования в коде;
- позволяет заменять объекты моками в тестах;
- меньшая связность в программе.
##### зачем нужен пустой интерфейс?
- для дженериков;
- стандартная библиотека использует его.
##### сколько места занимает интерфейс (а сколько слайд строк)?
Поле interface.data может содержать одно машинное слово, чаще всего оно занимает 8 байт. Но, к примеру, слайс []string займёт уже 24 байта: одно слово на указатель на массив, лежащий внутри слайса; одно слово на длину; и одно слово на оставшуюся вместимость массива (capacity). Но как Go должен втиснуть эти 24 байта в необходимые 8? Используя очень старый трюк, а именно **косвенную адресацию**. Да, слайс []string занимает 24 байта, но вот указатель на слайс []string — уже опять 8.

## Channels
> [Как на самом деле устроены каналы в Golang?](https://www.youtube.com/watch?v=ZTJcaP4G4JM)

 > паттерны: fan-in, fan-out, tee, done channel, done channel with struct, pipeline

### как отвечать
> **Каналы** — это способ безопасной коммуникации между goroutines.
> **Под капотом:**
> - очередь (buffer)
> - списки ожидающих goroutines
> - mutex
> **Небуферизированный** канал блокирует sender и receiver, пока обе стороны не готовы.

### Частые вопросы:
> - что будет при записи в nil канал? → **блокировка навсегда**
> - чтение из закрытого? → zero value + ok=false
> - запись в закрытый? → panic

### Mutex vs Channels?
> - Mutex лучше для защиты состояния (shared memory),  
> - Channels — для передачи данных (communication).
> В большинстве production случаев mutex проще и быстрее.

### как устроены каналы?
- кольцевой буфер (если буферизован), очередь пишущих и ожидающих горутин, мьютекс;
```go
type hchan struct {
   qcount   uint           // общение число элементов в очереди
   dataqsiz uint           // размер кольцевого буфера
   buf      unsafe.Pointer // указатель на элементы dataqsiz
                           // structure of buf: [ring buffer](https://ru.wikipedia.org/wiki/%D0%9A%D0%BE%D0%BB%D1%8C%D1%86%D0%B5%D0%B2%D0%BE%D0%B9_%D0%B1%D1%83%D1%84%D0%B5%D1%80)
   elemsize uint16 // размер элементов
   closed   uint32 // канал закрыт или нет
   elemtype *_type // тип элементов
   sendx    uint   // индекс отправляющих горутин в буфере
   recvx    uint   // индекс получающих горутин в буфере
   recvq    waitq  // список читающих горутин
   sendq    waitq  // список пишущих горутин

   // блокировка защищает все поля в hchan
   lock mutex
}
```

## Maps

### как отвечать
> **Map** — это хеш-таблица с бакетами.  
> При росте происходит **incremental resiz e**.
> **Не потокобезопасна** → нужен mutex.
### База
> есть два основных метода борьбы с коллизиями в мапах: **метод цепочек** и **метод открытой адресации**.
> в мапах в любом случае будут **коллизии**, потому что мы отображаем большое множество в малое.
### Важно:
- iteration order random
- concurrent write → panic
### Старая мапа
- просто **хэш-таблица**, массив бакетов, каждый бакет – таблица из 8 пар ключ-значение
- используется **метод цепочек**
- для значения вычисляется хэш с помощью **хэш-функции** и таким образом вычисляется номер бакета
- важно понимать, что могут возникнуть **коллизии** - если хэш-функция вернула одинаковые значения для разных ключей
**overflow bucket**
- при коллизиях в бакете значения складываются в таблицу/массив и в конце нее есть ссылка на **overflow bucket** (таким образом, другой бакет может лежать вообще в другой области памяти, что будет замедлять обращение к нему)
**эвакуация данных**
- если **load factor** превышен (среднее число элементов в бакете превышает 6.5, а заполненность мапы >80%), то происходит **рост мапы** X2
- эвакуация данных происходит не сразу, а инкрементально при каждой записи/удалении ключа
**tophash **
- помимо массива ключей и массива значений есть массив **tophash** (верхние 8 бит ключа) – это оптимизация для случаев, когда ключ очень большой (сравниваются сначала tophash-и, затем сами ключи)
### Новая мапа (swiss table)
> подробнее здесь – https://vkvideo.ru/video-139172865_456239348
- проблемы старой мапы:
	- **прыжки по памяти и промахи кэша** при сканировании цепочек
	- **мапа после роста не освобождает память** под раздутые бакеты
	- увеличение словаря приводит к **переиндексации всего**
- используется **метод с открытой адресацией**
**h1 и h2 части хэша ключа +SIMD операция** 
- от ключа строится хэш из **64 бит**, **h1=57 бит** и **h2=7 бит**
	- **первый h1** используется для поиска группы для ключа среди всех групп
	- **второй h2** для выбора ключа внутри группы + 1 бит - это **бит заполненности**
- вместо бакетов у нас группы, если для элемента в группе нет места, то кладем в следующую (при поиске также)
- **группа** – 8 слотов по 8 бит
**SIMD операции**
- single instruction - multiple data
- благодаря ним на уровне процессора можно за одну операцию найти слоты-кандидаты на искомый ключ – сравнение будет идти побайтово с 8 битами h2
**рост swiss таблиц**
- load factor = 7/8, то есть не менее 1/8 свободного слота в группе
- сначала рост таблицы в 2 раза (x2  кол-во групп), перенос данных сразу
- максимум 1024 слота (или 128 групп), далее переходим к концепции подтаблиц
- при переполнении – сплит на 2 swiss таблицы
**extendible hashing** – расширяемое хэширование
- снижает накладные расходы при росте таблицы и минимизирует перераспределение элементов за счет постепенного расширения таблицы
- идея в том, что когда данных много, то делаем много swiss таблиц вместо одной – все это называется директорией
- директория состоит из 8 swiss tables -> 2^3 = 8 -> используем три бита для определения таблицы внутри директории, далее h1 и h2 для точного определения места ключа
- важно понимать разницу global depth / local depth, что когда таблица делится на две, то local depth увеличивается и начинаем смотреть доп бит в ключе h1
- global depth = max(local depth), количество ячеек в dir (окно в h1) равно степени двойки

## Graceful Shutdown

> [Graceful Shutdown в Go на практике](https://habr.com/ru/articles/908344/)

### Нужно:
> - остановить приём трафика
> - завершить текущие задачи
> - использовать context + timeout

По умолчанию рантайм Go завершает приложение при получении сигналов `SIGTERM` , `SIGINT` или `SIGHUP`.
```go
func main() {
  signalChan := make(chan os.Signal, 1)
  signal.Notify(signalChan, syscall.SIGINT, syscall.SIGTERM)

  // Setup work here
  <-signalChan

  fmt.Println("Received termination signal, shutting down...")
}
```
`signal.Notify` указывает рантайм Go передавать сигналы в канал, вместо поведения по-умолчанию. При помощи этого мы обрабатываем сигналы самостоятельно и наше приложение не завершается автоматически.

С Go 1.16 можно сделать немного проще с `signal.NotifyContext` , который связывает обработку сигнала и контекст отмены:
```go
ctx, stop := signal.NotifyContext(context.Background(), syscall.SIGINT, syscall.SIGTERM)
defer stop()

// Setup tasks here

<-ctx.Done()
```
Важно знать сколько времени есть у приложения на завершение после получения сигнала. К примеру в Kubernetes **по-умолчанию это 30 секунд**, который можно переопределить в поле `terminationGracePeriodSeconds`

Особенно актуально для `readiness` проверки в Kubernetes, так как останавливающийся `pod` может все еще получать траффик, даже если другие end-points не готовы.

> Проверка `readiness` определяет, когда контейнер готов принимать траффик. Система периодически проверяет контейнер через настроенные HTTP запросы, TCP коннекты или запуск команд. Если проверка не проходит (в действительности можно настроить счетчик ошибок), Kubernetes удаляет `pod` из сервиса, чтобы контейнер не получал траффик, пока снова не станет готовым к работе.
```yml
readinessProbe:
  httpGet: 
    path: /healthz
    port: 8080
  periodSeconds: 5
```
В любом случае, `Shutdown` возвращает управление после того, как сервер полностью остановил обработку входящих запросов. По этой причине, ваши обработчики должны быть быстрыми и учитывать контекст. Иначе сервер может прервать их работу на середине, а это приводит к проблемам неконсистентности - частичная запись, потеря всех или части данных, открытые транзакции, повреждение данных.

> Ключевой принцип корректного завершения работы остается практически одинаковым для всех систем: **останавливаем прием данных (запросы, сообщения, сигналы), даем текущим операциям время на завершение с заранее определенным периодом.**

### выравнивание
- **зачем нужно?**
	- современные процессоры читают память блоками и чтобы одно значение читалось из одного блока, а не нескольких (можно представить массив из элементов по 3 байта без выравнивания, отдельные элементы будут читаться из неск блоков)
- **как выравниваются массивы?**
	- по размеру элемента (int32 - 4, int64 - 8, bool - 1)
- **как выравниваются слайсы?**
	- слайс - структура 24 байта, выравнивание по 8 байт (одно машинное слово)
- **как выравниваются строки?**
	- строка (указатель + размер) - 16 байт, выравнивание по 8 байт
- **выравнивание на 32/64 битных машинах?**
	- int будет весить либо 4 байта, либо 8
- **выравнивание по 4 или по 8 байт?**
	- если есть поля в структуре по 8 байт, то будет выравнивание по 8, а если есть только по 4 байта поля, то будет и выравнивание по 4 байта
### интернирование строк
- пакет unique - unique.Make(str)
- делает дедуплицирование строк
- делает последующее сравнение строк эффективнее и быстрее
#### **барьеры памяти**
- **есть несколько видов барьеров:**
	- **полный барьер**
	- барьер чтения
	- барьер записи
	- барьер acquire (LoadLoad + LoadStore) - только внутрь/вниз барьер барьер release (StoreStore + LoadStore) - только наружу/вверх барьер
- соотв компилятор и процессор **НЕ смогут менять местами операции до и после барьера** (для полного барьера; для барьера чтения они смогут переставлять только операции записи; для барьера записи смогут менять только операции чтения)
- для атомиков в go используется **full barier**

## Другое полезное

### Peek vs Read в go1.26
> peek – zero-copy версия read, возвращает слайс на нежележащий слайс
> peek грубо говоря просто достает ссылку на некий буфер без копирования, как слайс ссылается на исходный массив, а read копирует данные в наш буфер, который мы в него передадим
> peek не сдвигает внутренний массив
> peek будет менять нижележащий массив, а после read буфер лично твой и исходный буфер не меняется

### Process Handle в go1.26
##### Проблема
```plain
 Запускаем процесс → PID 1234
2. Процесс завершается
3. Ядро через 1ms выдаёт PID 1234 новому процессу
4. Мы делаем proc.Kill() — убиваем совершенно другой процесс!
```
Это **race condition**, классическая проблема Unix.
ядро Linux 5.4+ ввел **pidfd** — файловый дескриптор, который **жёстко привязан к конкретному процессу**.
в **go раньше**: Go уже использовал pidfd внутри `os.Process` (если ОС поддерживала), но **не давал доступ** к этому дескриптору наружу.
в **go 1.26 добавили**: ```gofunc (p *Process) WithHandle(f func(handle uintptr)) error```
```go
// А так — безопасно:
proc.WithHandle(func(handle uintptr) {
    // handle гарантированно валиден ВНУТРИ этой функции
    // После return — handle больше не твой
})
```
### Signal as cause — какой сигнал отменил контекст

##### Что было раньше
```go
ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt)
defer stop()

<-ctx.Done()

fmt.Println(ctx.Err())        // "context canceled"
fmt.Println(context.Cause(ctx)) // "context canceled" ← НЕ ПОНЯТНО, ПОЧЕМУ!
```
**Проблема:** если контекст отменился — не ясно, какой именно сигнал пришёл. Было это SIGINT, SIGTERM, SIGHUP?
Теперь `context.Cause(ctx)` возвращает **конкретный сигнал**:
```go
<-ctx.Done()

fmt.Println(ctx.Err())          // "context canceled"
fmt.Println(context.Cause(ctx)) // "signal: interrupt" ← ВИДНО!
```
##### Тип ошибки
```go
// Внутри пакета signal
type signalError string

func (e signalError) Error() string { return string(e) }
```
##### Практика

| Сценарий                           | Как использовать                                        |
| :--------------------------------- | :------------------------------------------------------ |
| Graceful shutdown с разной логикой | Проверяем cause: SIGTERM → drain, SIGINT → быстрый exit |
| Метрики                            | Логируем, какой сигнал чаще приходит                    |
| Тесты                              | Проверяем, что именно SIGTERM отменил контекст          |

Теперь можно различать SIGINT/SIGTERM/SIGHUP в graceful shutdown
```go
ctx, stop := signal.NotifyContext(
    context.Background(),
    syscall.SIGINT,
    syscall.SIGTERM,
)
defer stop()

<-ctx.Done()

switch context.Cause(ctx).Error() {
case "signal: interrupt":
    log.Println("SIGINT — быстрый выход")
case "signal: terminated":
    log.Println("SIGTERM — graceful shutdown")
}
```
