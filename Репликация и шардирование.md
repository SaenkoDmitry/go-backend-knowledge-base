### **📌 RPO (Recovery Point Objective) – “насколько назад откатимся”**
> - сколько данных ты готов потерять
> - максимальный допустимый “разрыв” по данным
> 
> **Например**, RPO = 5 минут
> **👉 значит:**
> - ты можешь потерять **до 5 минут данных**;
> - всё, что записано после последнего бэкапа/реплики — может исчезнуть
> - **RPO ≈ 0 => синхронная репликация** – достигается при sync replication или quorum write

### **📌 RTO (Recovery Time Objective) – “как быстро поднимемся”**
> сколько времени система может быть недоступна
> сколько времени система может быть down
> 
> **Например**, RTO = 10 минут
> **👉 значит:**
> - у тебя есть 10 минут, чтобы:
    - поднять реплику
    - переключить трафик
    - восстановить сервис
> **RTO ≈ 0 => hot standby / active-active**


### **📌 Trade-off RTO/RPO**
Чем меньше RPO/RTO → тем дороже система

### **📌 Hot Standby**
> резервный инстанс уже запущен, синхронизирован, и готов мгновенно принять нагрузку
##### **Как работает**
- данные реплицируются (sync или async)
- standby постоянно обновляется
- при падении primary:
	- происходит failover
	- standby становится новым primary

### **📌 Active-Active**
> несколько инстансов **одновременно обрабатывают запросы**
```
Client → Load Balancer → [Node A, Node B, Node C]
```
##### **Как работает**
- все ноды активны
- трафик балансируется
- данные должны быть:
    - синхронизированы
    - или шардингованы

### **📌 Разница Hot Standby vs Active-Active**
```
Hot Standby:  
1 активный → 1 ждёт  
  
Active-Active:  
N активных → все работают
```

### **📌 Failover**
> это автоматическое (или ручное) переключение с отказавшего компонента на резервный.

**Было:**
```
Primary (работает)  
Replica (ждёт)
```
**Стало:**
```
Primary (упал)  
Replica → становится новым Primary
```

## **Репликация**

> Репликация увеличивает устойчивость и масштаб чтения

- [[Репликация и шардирование#**Для чего**|Для чего]]
- [[Репликация и шардирование#**Read / Write path**|Read / Write path]]
- [[Репликация и шардирование#**Модели репликации**|Модели репликации]]
	- [[Репликация и шардирование#**Primary-Replica**|Primary-Replica]]
	- [[Репликация и шардирование#Multi-Primary]]
	- [[Репликация и шардирование#**Leaderless**|Leaderless]]

### **Для чего**
> - повышения availability
> - масштабирования read-нагрузки
> - уменьшения RPO

### **Read / Write path**  

##### **Write path (Primary-Replica)**  
> Client → Leader → Replicas → Ack → Client  
```
- sync → ждём replicas → latency ↑ consistency ↑  
- async → быстро → риск потери данных ↑ (RPO)  
```
##### **Read path** 
> Client → Replica
```
- leader read → strong consistency  
- replica read → faster, но возможен stale (replica lag)
```

### **Модели репликации**

#### **Primary-Replica**
> Один write-leader, несколько read-replicas. Хорошо для read-heavy нагрузок и понятного операционного управления.
> **Replica lag** и асимметрия чтение/запись;
> **failover** должен быть автоматизирован и протестирован.
###### **Когда использовать**
> - read-heavy workload
> - простота

#### **Multi-Primary**
> Запись в нескольких регионах/нодах, ниже latency для geo-distributed writes.
> Конфликты и сложная merge-логика;
> Высокий порог сложности для команд и платформы.

#### **Leaderless**
> Кворумная запись/чтение и высокая доступность для distributed KV/NoSQL сценариев.
> Кворум – то есть для записи нужно W подтверждений от нод, для чтения R подтверждений от нод, а всего реплик должно быть N (W + R > N).
> Сложнее reasoning о консистентности, read repair, hinted handoff и tombstones.
> eventual consistency по умолчанию.
###### **Когда использовать**
> - высокая доступность
> - tolerate partitions

> - **read repair** — при чтении исправляем расхождения
> - **hinted handoff** — временно храним запись за недоступную ноду

### **Модели шардирования**

- **Hash-based sharding** – равномерное распределение, но сложнее range-запросы
- **Directory-based sharding** – гибкое управление расположением данных, требует надежного metadata service
- **Range-based sharding** – эффективно для диапазонных запросов, но есть риск hot ranges
- **Geo/tenant sharding** – удобно для data sovereignty и изоляции клиентов, но усложняет cross-tenant операции

### **Проблема Hot shards**
> пример: user_id = celebrity → все запросы в один shard

### **Rebalancing**
- Consistent hashing и virtual nodes
- добавление/удаление нод → перераспределение данных

### **Миграции и безопасность**  
> как безопасно перемещать данные между шардами/нодами
- backfill через throttled pipelines
- checksum verification
- pause/resume, rollback, progress

#### **backfill через throttled pipelines**
> **Backfill** — это копирование “старых” данных в новое место

**Пример**
Ты добавил новую ноду → нужно переложить часть данных:
```
Shard A → Shard B
```
**👉 нельзя просто “влить всё сразу”, иначе:**
- убьёшь прод (CPU / IO / network)
- увеличишь latency
**Поэтому:**
> **throttled pipelines** — копируем данные постепенно, с ограничением скорости

#### **Checksum verification**
> Проверка, что данные после миграции **не испортились**

**Как:**
- считаешь hash (например CRC / MD5)
- сравниваешь:
```
старый shard == новый shard ?
```
👉 если нет → retry / алерт

## **Согласованность**

> Consistency — это то, какие гарантии дают репликация и шардирование при чтении данных

### **Модели consistency**
- **Strong consistency** — читаешь последнее записанное значение
- **Eventual consistency** — данные со временем сходятся
- **Read-your-writes** — пользователь видит свои записи
- **Monotonic reads** — не читаешь “назад во времени”

### **Зависит от:**
- replication strategy
- quorum
- read path

### **CAP: Consistency, Availability, Partition tolerance**
при network partition выбираем между consistency и availability. Так как устойчивость к разделению, как правило, обязательна.
> CP: etcd, Zookeeper, HBase — жертвуют availability
> AP: Cassandra, DynamoDB — жертвуют consistenc

## **Trade-offs**

### **Replication**:
- sync → consistency ↑ latency ↑
- async → latency ↓ риск потерь ↑

### **Sharding:**
- больше шардов → масштаб ↑ сложность ↑
- меньше шардов → проще, но bottleneck

### **Leader-based:**
- проще обеспечивать strong consistency
- single write bottleneck

### **Leaderless:**
- высокая availability
- сложная consistency

## **Leader election в Raft**

- [[Репликация и шардирование#**Роли в Raft**|Роли в Raft]]
- [[Репликация и шардирование#**Как проходит election**|Как проходит election]]
- [[Репликация и шардирование#**Где это используется**|Где это используется]]
- [[Репликация и шардирование#**Как работает на примере репликации данных**|Как работает на примере репликации данных]]
- [[Репликация и шардирование#**Связь с failover**|Связь с failover]]

### **Роли в Raft**
```
Follower (по умолчанию все ноды) → (кандидат) → Leader (один)
```
##### **Follower**
- по умолчанию все ноды — followers
- просто принимают данные от лидера
- сами ничего не решают
##### **Candidate**
- нода, которая решила: “лидера нет, пора выбрать”
##### **Leader**
- принимает все записи
- рассылает их остальным

### **Как проходит election**

##### **Шаг 1. Пропал лидер**
Follower ждёт heartbeat от лидера:
```
Leader → "я жив" → Followers
```
Если heartbeat нет (timeout):
👉 нода думает: “лидер умер”
##### **Шаг 2. Нода становится Candidate**
```
Node B → Candidate
```
- увеличивает term (номер эпохи)
- голосует за себя
- рассылает запросы:
```
"голосуй за меня"
```
##### **Шаг 3. Голосование**
**Правила:**
- каждая нода голосует **1 раз за term**
- побеждает тот, кто получил **большинство (quorum)**
```
из 3 нод → нужно 2 голоса
```
##### **Шаг 4. Появляется лидер**
```
Node B → Leader
```
И сразу начинает:
```
"я лидер, держите heartbeat"
```
#### **Почему это работает**
> 👉 Ключевая идея — **quorum (большинство)**
> - два разных лидера не могут получить большинство одновременно
> - значит → **нет split-brain**

> **Split-brain** — ситуация, когда два узла считают себя лидером  
> → решается через quorum / fencing / consensus (Raft)
#### **Что если 2 кандидата одновременно**
```
Node A и Node B оба стали Candidate
```
- они делят голоса
- никто не получает majority
- ждут случайный timeout
- пробуют снова
👉 поэтому Raft использует **randomized timeout**

### **Где это используется**
- etcd
- Consul
- Kubernetes (через etcd)

### **Как работает на примере репликации данных**
Leader:
1. принимает запись
2. отправляет её followers
3. ждёт подтверждения от большинства
4. только потом считает запись committed
👉 это важно для consistency

### **Связь с failover**
- лидер упал → election
- новый лидер выбран → это и есть **failover**
- время election → часть твоего **RTO**

## **Как масштабировать базу данных**

> Масштабирование базы обычно делается в несколько этапов, в зависимости от узкого места.

#### **Кратко**
> Я начинаю с vertical scaling, затем добавляю репликацию для read scaling и availability. Когда упираюсь в write throughput — перехожу к шардированию. Дальше важно управлять consistency, read/write path и операционными вещами вроде rebalancing и миграций.

- [[Репликация и шардирование#**1. Сначала — вертикальное масштабирование**|1. Сначала — вертикальное масштабирование]] – накидываем CPU, RAM, SSD
- [[Репликация и шардирование#**2. Добавляем репликацию**|2. Добавляем репликацию]] – читаем с реплик, пишем в leader
- [[Репликация и шардирование#**3. Масштабируем запись через шардирование**|3. Масштабируем запись через шардирование]] – hash / range based etc
- [[Репликация и шардирование#**4. Работаем с consistency**|4. Работаем с consistency]] – latency vs consistency
- [[Репликация и шардирование#**5. Оптимизируем read/write path**|5. Оптимизируем read/write path]] – sync vs async
- [[Репликация и шардирование#**6. Решаем операционные проблемы**|6. Решаем операционные проблемы]] – rebalancing, миграции
- [[Репликация и шардирование#**7. (Опционально) geo-scale**|7. (Опционально) geo-scale]]

#### **1. Сначала — вертикальное масштабирование**
> Увеличиваем ресурсы одной ноды (CPU, RAM, SSD)
```
- просто
- быстро
- но есть предел
```

#### **2. Добавляем репликацию**
> Чтобы повысить доступность и масштабировать чтение
```
- Primary → Replica
- читаем с реплик
- пишем в leader
```
**Trade-off:**
- replica lag → возможны stale reads

#### **3. Масштабируем запись через шардирование**
> Делим данные между нодами
```
- hash-based → равномерно
- range-based → удобно для range queries
```
**Проблемы:**
- hot shards
- rebalancing

### **4. Работаем с consistency**
> Выбираем баланс между latency и consistency
```
- strong → медленнее, но точно
- eventual → быстрее, но возможны расхождения
```

#### **5. Оптимизируем read/write path**
- read с реплик → быстрее
- write:
    - sync → безопасно
    - async → быстрее

#### **6. Решаем операционные проблемы**
- rebalancing при росте
- миграции (backfill, checksum)
- мониторинг lag и hot shards

#### **7. (Опционально) geo-scale**
> Если система глобальная:
```
- multi-region
- multi-primary или leaderless
```

## **Как шардировать базу данных на 100M пользователей**

#### **Кратко**
> Я бы начал с hash-based sharding по user_id, так как это даёт равномерное распределение и простоту масштабирования.
> Чтобы уменьшить стоимость rebalancing при добавлении нод, я бы использовал consistent hashing с virtual nodes.
> Каждый shard я бы сделал replication group (leader + replicas), чтобы обеспечить availability и масштабирование чтения.
> Дальше я бы учёл проблемы hot shards — например, для популярных пользователей добавил бы caching или key salting.
> Для более гибкого управления, например при миграциях или неравномерной нагрузке, можно использовать directory-based routing.
> И важно, что rebalancing я делаю онлайн: через throttled backfill, с валидацией и постепенным переключением трафика.

#### **Шаг 0. Уточнения**
- какой основной access pattern?
    - по `user_id`?
    - по `email`?
- есть ли range-запросы?
- write-heavy или read-heavy?
- есть ли “горячие” пользователи? (celebs)
- нужен ли multi-region?

#### **Шаг 1. Базовое решение**
> Я начну с hash-based sharding по user_id
```
shard = hash(user_id) % N
```
**Почему:**
- равномерное распределение ✔
- просто ✔
- хорошо масштабируется ✔

#### **Шаг 2. Проблемы**

**🔴 1. Hot users**
> celebrity → много запросов → один shard перегружен

**🔴 2. Rebalancing**
> добавили ноду → почти все ключи переехали

**🔴 3. Range queries**
> “все пользователи от A до B” → плохо

#### **Шаг 3. Улучшения**

###### **✅ 1. Consistent hashing + virtual nodes**
> чтобы минимизировать миграции
```
hash(user_id) → ring → vnode → shard
```
**Плюсы:**
- при добавлении ноды → двигается только часть данных
- балансировка лучше

###### **✅ 2. Репликация поверх шардов**
> каждый shard = replication group
```
Shard 1 → [Leader + Replicas]Shard 2 → [Leader + Replicas]
```
**👉 это важно:**
- availability
- read scaling

###### **✅ 3. Обработка hot shards**
Если есть “горячие” ключи:

**варианты:**
- **key salting**
```
user_id → user_id + random_suffix
```
- **logical partitioning**
> разбиваем одного пользователя на несколько partition’ов
- **cache (очень часто!)**
> горячие данные уносим в Redis

###### **✅ 4. Directory-based routing (если нужно больше контроля)**
> вместо hash → используем mapping
```
user_id → shard_id (в metadata service)
```
**когда нужно:**
- uneven distribution
- VIP users
- миграции

###### **✅ 5. Rebalancing стратегия**
> “Я делаю online rebalancing”
- backfill с throttling
- dual writes (временно)
- переключение трафика
- checksum

#### **Шаг 4. Если система глобальная**
> “Если есть multi-region…”

**варианты:**
1. Geo-sharding
```
EU users → EU cluster  US users → US cluster
```
2. Multi-primary (осторожно)
	- ниже latency
	- но конфликты

#### **Дополнительные вопросы**

**❓ “А если shard переполнился?”**
→ split shard / добавить ноды / rehash

**❓ “Как делать cross-shard queries?”**
→ scatter-gather / ограничивать / secondary index

**❓ “Как делать транзакции?”**
→ избегать / eventual consistency / 2PC (дорого)

**❓ “Как найти пользователя?”**
→ deterministic routing (hash) или directory
