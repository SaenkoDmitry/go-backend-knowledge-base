## **Балансировка L4 и L7**

### **Виды балансировки**
**L4** (транспортный слой, тупой прокси) и **L7** (слой приложения, умный прокси)

### **Как выбрать: L4 или L7?**

> - Часто каскад: L4 (edge) → L7 (internal) → backend

**L4 балансировка**
```plain
Клиент ──TCP SYN──► [L4 Балансировщик] ──TCP SYN──► Backend:8080

Решение принимается ПЕРЕД установкой полного TCP-соединения с бэкендом
или ПОСЛЕ (в зависимости от режима), но ВСЕГДА на основе:
- Source IP
- Destination IP:Port
- TCP-флаги (SYN, ACK и т.д.)
```

**L7 балансировка**
```plain
Клиент ──HTTP GET /api/v1/users──► [L7 Балансировщик] ──HTTP GET──► Backend
                                    (Nginx/HAProxy/Envoy)
1. Принимает полный HTTP-запрос (буферизует headers/body)
2. Парсит Host, Path, Headers, Cookies
3. Может проверить JWT, rate limit по user-id
4. Открывает НОВОЕ соединение к backend (или reuse keep-alive)
5. Может модифицировать запрос (добавить X-Forwarded-For)
```

**Сравнение L4/L7**

|                    | L4 (тупой прокси)                       | L7 (умный прокси)                                                             |
| ------------------ | --------------------------------------- | ----------------------------------------------------------------------------- |
| протокол           | TCP/UDP                                 | HTTP/gRPC                                                                     |
| маршрутизация      | IP+port                                 | path/host/header/cookie                                                       |
| гибкость           | ниже: обычно hash/leastconn/round-robin | выше: обычно canary, A/B, sticky session, rate limiting, policy-based routing |
| производительность | меньше CPU/latency overhead             | больше стоимость на запрос за счет доп логики, но зато точнее контроль        |
| типовые сценарии   | БД, Redis, MQTT                         | API Gateway, web приложения, gRPC mesh, product traffic policies              |
### Почему два уровня?

- L4 на границе (edge) — быстро откидывает атаки, держит миллионы idle-соединений (WebSocket)
- L7 внутри — умный routing, но меньше нагрузка, так как L4 уже отфильтровал мусор
### Вопросы:
- **Почему L7 медленнее L4?**
	- L7 терминирует TCP-соединение клиента (полный handshake), буферизует и парсит HTTP-запрос, потом может открывать новое соединение к backend. L4 может работать в режиме **DSR (Direct Server Return)** или **NAT**, где handshake проходит через балансировщик, но данные идут напрямую backend→клиент, минуя балансировщик.
- **Как сделать sticky sessions в L7?**
	- Cookie-based: балансировщик ставит cookie `SRV=server1`, следующие запросы идут туда же
	- IP-based: хеш от source IP (менее надёжно)
    - **Важно**: L7 sticky sessions точнее, потому что видит cookies, а не просто IP
- **L4 или L7?**
	- Зависит от требований. L4 для производительности и прозрачности, L7 для интеллектуального routing. Часто используют оба уровня в каскаде.
- **Как узнать реальный IP клиента за L7 балансировщиком?**
	- `X-Forwarded-For`, но нужно валидировать (взять ПОСЛЕДНИЙ доверенный прокси, а не первый, иначе клиент подделает).
		```go
		// L7 балансировщик (Nginx/Envoy) обычно добавляет заголовки
		// Go-код должен доверять или проверять их
		func handler(w http.ResponseWriter, r *http.Request) {
		    // L7 прокси передал реальный IP клиента
		    realIP := r.Header.Get("X-Forwarded-For")
		    if realIP == "" {
		        realIP = r.RemoteAddr // напрямую, без прокси
		    }
		    
		    // Если L7 терминировал TLS — проверяем proto
		    proto := r.Header.Get("X-Forwarded-Proto")
		    if proto == "https" {
		        // клиент пришёл по HTTPS, хотя мы видим HTTP
		    }
		}
		```

## **Алгоритмы балансировки**

> Для **system design interview** эта тема особенно полезна тем, что переводит разговор из уровня "какой алгоритм взять" в уровень "**какой профиль трафика** мы на самом деле защищаем и что сломается при неверном выборе".
### **Виды**
- [[Сети#Round Robin|Round Robin]] – по кругу
- [[Сети#Weighted Round Robin|Weighted Round Robin]] –  если сервера различной мощности
- [[Сети#Least Connections|Least Connections]] – самый свободный
- [[Сети#IP Hash|IP Hash]] – sticky sessions
- [[Сети#Consistent Hashing|Consistent Hashing]] – оптимизированная перебалансировка при добавлении/удалении серверов (перераспределяются ~1/N данных)
#### **Важно**
- [[Сети#Как выбирать алгоритм?|Как выбирать алгоритм?]]
- [[Сети#Поведение при burst нагрузке|Поведение при burst нагрузке]]

### **Round Robin**

> Просто по очереди отправляет запросы на сервера. A → B → C → A → B → C

#### **✅ Когда подходит**
- одинаковые сервера
- stateless сервисы
- равномерный трафик
#### **❌ Проблемы**
- не учитывает:
    - latency
    - load
    - разную мощность серверов
#### **⚡ Burst нагрузка**
- плохо адаптируется

### **Weighted Round Robin**

> **Сервера получают разный вес:**
> A (3) → B (1)  
> A A A B A A A B ...

#### **✅ Когда подходит**
- разная мощность серверов
#### **⚡ Burst нагрузка**
- всё ещё не adaptive

### **Least Connections**

> Отправляем запрос туда, где **меньше активных соединений**.

#### **✅ Когда подходит**
- long-lived connections (WebSocket, streaming)
- разная длительность запросов
#### **❌ Проблемы**
- не учитывает:
    - CPU usage
    - реальную нагрузку (один запрос может быть тяжелее 100 лёгких)
#### **⚡ Burst нагрузка**
- лучше, чем Round Robin
- но может “обмануться”, если:
    - есть много быстрых коротких запросов
    - и один медленный тяжелый
#### **Реализация**
```go
type Server struct {
    Addr        string
    Connections int
}

func pickLeast(s []Server) *Server {
    var best *Server
    for i := range s {
        if best == nil || s[i].Connections < best.Connections {
            best = &s[i]
        }
    }
    return best
}
```

### **IP Hash**

> простой способ sticky sessions, но плохо масштабируется и неустойчив к skew.

#### **✅ Когда подходит**
- нужна **sticky session**
- state хранится на сервере
#### **❌ Проблемы**
- плохой баланс при:
    - NAT (много клиентов за одним IP)
- при добавлении сервера:
    - почти всё перераспределяется
#### **⚡ Burst нагрузка**
- если один IP генерит burst → убивает один сервер
#### **Реализация**
```
hash(client_ip) % N
```

### **Consistent Hashing**

> Хешируем:
> - клиентов (или ключи)
> - серверы на кольце

> **🔥 Ключевая фишка:** 
> При добавлении сервера:
> - перераспределяется **~1/N данных**, а не всё

#### **✅ Когда подходит**
- кеши (Redis, Memcached)
- шардирование
- минимизация rebalancing

#### **❌ Проблемы**
- uneven distribution (решается virtual nodes)
- сложнее реализация

#### **⚡ Burst нагрузка**
- зависит от ключей:
    - hotspot → один сервер перегружен

#### **Реализация**
```go
hash := sha1(key)  
server := ring.FindClosest(hash)
```

### **Как выбирать алгоритм?**

> 👉 **нет универсального алгоритма**
> 👉 выбор = тип нагрузки + state + поведение системы

#### 📊 Если сервис stateless:
- start: Round Robin
- лучше: Least Connections / Least Latency
#### 📊 Если есть state (session):
- IP Hash (простой)
- Consistent Hash (правильный)
#### 📊 Если кеш / шардирование:
- только Consistent Hash
#### 📊 Если нагрузка неравномерная:
- Least Connections
- или adaptive (latency-based)
#### 📊 Если разные серверы:
- Weighted algorithms
### **Поведение при burst нагрузке**

| Алгоритм          | Поведение              |
| ----------------- | ---------------------- |
| Round Robin       | ❌ тупо продолжает      |
| Least Connections | ⚠️ частично помогает   |
| IP Hash           | ❌ hotspot убивает ноду |
| Consistent Hash   | ⚠️ зависит от ключей   |
| Least Latency     | ✅ лучший               |

## **NGINX**

### Что это: **Обратный прокси + веб-сервер**.

### **Кратко**
- Async event-driven (epoll), master + workers
- Web-server + reverse proxy + L4/L7 balancer
- Zero-downtime reload через master/worker handoff (новые worker'ы с новым конфигом, старые worker'ы не принимают новые соединения, но завершают текущие)
- Keep-alive reuse соединений к backend'ам

### **Подробнее**
#### **Может работать как:**
- **Reverse proxy** — проксирует запросы к backend'ам
- **Forward proxy** — редко используется, но умеет
- **Web-server** — отдаёт статику напрямую (html, css, js, картинки)
- **Load balancer** — L7 (http) и L4 (stream/tcp)
#### **Задачи:**
- **балансировка** запросов между несколькими внутренными сервисами (иногда это делает балансировщик)
- **безопасность** - чтобы предотвратить доступ к ресурсу из интернета, защитить от атак
- **шифрование** TLS (transport layer security over IP), SSL (secure socket layer over IP)
- **кэширование** контента веб страниц или определенных ресурсов для снижения нагрузки на веб сервисы

> проще говоря, он позволяет отдавать статические ресурсы сервера (картинки, файлы и тд), в более общем (сложном) случае он позволяет еще и выполнять сжатие, кэширование, балансировку и тд

#### **Архитектура**
```plain
# главная идея в том, что мы имеем мастер процесс и набор воркеров (по дефолту установлены по числу процессорных ядер) и эти воркеры обрабатывают одновременно до 1024 соединений

Master process (1)
├── Worker 1 ──► event loop (epoll/kqueue) ──► обрабатывает 10K+ соединений
├── Worker 2 ──► event loop                              (async non-blocking)
├── Worker 3 ──► event loop
└── Worker N   (N = CPU cores по умолчанию)
```
#### **Особенности**
> - **Ключевой момент**: Nginx использует **async event-driven** модель, а не thread-per-connection.
> - Каждый worker — **один поток**
> - Внутри — event loop на `epoll` (Linux) / `kqueue` (BSD/macOS)
> - **Нет блокировки** на I/O: один worker держит 10K+ соединений
> - Event loop на epoll, аналогично тому, как Go runtime мультиплексирует горутины на потоки

более подробно - [https://habr.com/ru/post/260065/](https://habr.com/ru/post/260065/)
#### **Что спросят на собесе?**

- **Почему Nginx быстрее Apache?**
	- Apache использует process-per-connection или thread-per-connection (mpm_prefork, mpm_worker). Nginx — async event-driven с epoll. Меньше переключений контекста, меньше памяти на соединение.
- **Что происходит при reload?**
	- Master читает новый конфиг, порождает новых worker'ов. Старые worker'ы перестают принимать новые соединения, но завершают текущие. Zero-downtime deployment.

## **HaProxy**

**High Availability Proxy** — специализированный балансировщик. L4/L7 load balancer и proxy с расширенными возможностями.

### **Кратко**
- Специализированный load balancer (не веб-сервер)
- L4 и L7 изначально, мощные ACL
- Active health checks в бесплатной версии
- динамические изменения без reload
- stick-table для сложного rate limiting
### **Подробнее**
##### **Архитектура (отличие от Nginx)**
```plain
HAProxy process (обычно 1, можно multi-process)
├── Listener threads (привязаны к CPU cores)
│   └── Event loop (epoll/kqueue) 
├── Health check threads
└── DNS resolution threads
```

> **Ключевое отличие от Nginx**: HAProxy — **single-process** (по умолчанию) или **multi-process** с shared memory. Нет master/worker иерархии. Весь конфиг в одном процессе.
> Многопоточность внутри одного процесса, аналогично Nginx worker'ам.

##### **Как он работает**
- **Event-driven** (как Nginx) — async non-blocking I/O
- **Zero-copy** где возможно
- **No buffering by default** — потоковая передача (streaming), минимальная задержка
- **Connection reuse** — `http-reuse aggressive` для keep-alive к backend'ам

##### **Почему HAProxy считается 'правильным' балансировщиком?**
> Он делает одну вещь и делает её идеально. Нет лишней функциональности веб-сервера. Меньше attack surface. Мощные health checks и runtime API из коробки.

##### **Когда выбрать Nginx, когда HAProxy?**
- Нужен веб-сервер + отдача статики + проксирование -> **Nginx**
- Чисто балансировка, максимальная гибкость -> **HAProxy**
- Runtime изменение конфига без reload -> **HAProxy**
- gRPC, HTTP/2, WebSocket + кэширование -> **Nginx**
- DDoS protection, сложный rate limiting -> **HAProxy**

## **Access Control List (ACL)**

ACL (Access Control List) — это **условия** (predicates), которые Nginx/HAProxy проверяет на основе входящего трафика. По сути — `if` в конфиге на то, каким клиентам какие доступы разрешены.

## **Forward Proxy vs Reverse Proxy**

### **Reverse Proxy (обратный прокси)**
```plain
Интернет ──► [Reverse Proxy: Nginx/HAProxy] ──► [Backend: Go/Python/Java]
                 203.0.113.1                         10.0.1.10:8080
                 
Клиент знает: "Я хочу example.com"
Reverse proxy знает: "example.com → 10.0.1.10:8080"
Backend не знает о клиенте напрямую — видит только reverse proxy
```
##### **Роли reverse proxy**
- **Скрывает backend'ы** (они в приватной сети)
- **Терминирует TLS** (клиент ↔ proxy шифровано, proxy ↔ backend может быть HTTP)
- **Балансирует** между backend'ами
- **Кэширует** ответы
- **Защищает от атак** (WAF, rate limiting)
### **Forward Proxy (прямой прокси)**
```plain
[Client: 192.168.1.50] ──► [Forward Proxy: Squid/ corporate proxy] ──► [Internet: example.com]
    корпоративная сеть          10.0.0.1:3128                           93.184.216.34:443
    
Клиент знает: "Я хочу example.com, но я обязан идти через прокси"
Forward proxy знает: "192.168.1.50 хочет example.com, я схожу за него"
Сервер example.com не знает о клиенте — видит только forward proxy
```
##### **Роли forward proxy**
- **Контроль доступа** в интернет из корпоративной сети ("блокируем Facebook")
- **Кэширование** — сотрудники качают одни и те же обновления Windows
- **Анонимизация** — сервер видит IP прокси, а не клиента
- **Обход блокировок** — клиент в Китае ходит через прокси в Европе
- **Логирование** — кто куда ходил (compliance)
### **Что спросят на собесе**

**В чём разница reverse и forward proxy?**
- Reverse proxy стоит перед серверами, защищает их, балансирует трафик. Forward proxy стоит перед клиентами, контролирует их доступ в интернет, кэширует, логирует.

**Может ли Nginx быть forward proxy?**
- Технически да, но это антипаттерн. Для forward proxy используют Squid, Privoxy, или специализированные корпоративные решения.

## **Global Server Load Balancing (GSLB)**

> **Про что это:** про балансировку между регионами и дата-центрами, а не только между инстансами внутри одного кластера.
> **Почему это важно:** локальный LB не решает задачи global latency и regional outage. Без GSLB пользователи могут ходить в далекий регион или застревать на деградировавшей площадке.
> **Для каких сценариев подходит:** multi-region продукты, глобальный B2C-трафик, требования по RTO/RPO, региональные compliance-ограничения и активный disaster recovery.

#### **Бывает нескольких типов:**

1) DNS-based GSLB
	- Авторитативный DNS выбирает регион по latency/geo/weight/health и возвращает ближайший endpoint.
2) Anycast
	- Один и тот же IP анонсируется из нескольких PoP/регионов; BGP направляет трафик в топологически ближайшую точку.

## Health checks

#### Active health checks
Балансировщик сам опрашивает health endpoint/TCP handshake. Можно задать interval, timeout, rise/fall и исключать деградировавшие инстансы до hard-failure.
#### Passive health checks
Решение о деградации принимается по реальному трафику: 5xx, timeout, reset rate, high latency. Это полезно, когда endpoint формально жив, но по факту сервис уже не держит нагрузку.

## **Service mesh**

**Service Mesh** — **инфраструктурный слой**, который управляет всем сетевым взаимодействием между микросервисами.
##### **Ключевая идея: вынести сетевую логику (retry, timeouts, circuit breaker, metrics, etc) из приложения в отдельный прокси-слой**.

> **в Kubernetes – Envoy, Istio**

> **Про что это:** про балансировку на уровне service-to-service трафика в микросервисной среде, где каждое внутреннее RPC - отдельное LB-решение.
> **Для каких сценариев подходит:** десятки/сотни сервисов, единые traffic policy, canary/traffic splitting, mTLS и необходимость централизованно управлять retries, outlier detection и locality failover.
> **Когда НЕ использовать**: 2-3 сервиса (overkill), ultra-low latency requirements (HFT), legacy without containers.

```plain
БЕЗ Service Mesh (традиционно):
┌─────────────┐     HTTP/gRPC      ┌─────────────┐
│  Service A  │ ─────────────────► │  Service B  │
│  (Go app)   │   (client-side LB) │  (Go app)   │
└─────────────┘                    └─────────────┘
     ↑                                  ↑
   В коде A: retry, timeout,           В коде B:
   circuit breaker, metrics...          просто слушает порт

Проблема: логика размазана по всем сервисам, на каждом языке своя реализация


С Service Mesh (sidecar pattern):
┌─────────────────────────────────┐     mTLS/gRPC     ┌─────────────────────────────────┐
│  Service A                      │                   │  Service B                      │
│  ┌─────────┐  localhost:15001  │                   │  localhost:15001  ┌─────────┐   │
│  │ App A   │ ◄──────────────►  │  ─────────────►   │  ◄──────────────► │ App B   │   │
│  │ (Go)    │                   │   (mTLS, LB,      │                   │ (Go)    │   │
│  └─────────┘                   │    retries...)      │                   └─────────┘   │
│  ┌─────────┐                   │                   │                   ┌─────────┐   │
│  │ Envoy   │                   │                   │                   │ Envoy   │   │
│  │(sidecar)│                   │                   │                   │(sidecar)│   │
│  └─────────┘                   │                   │                   └─────────┘   │
└─────────────────────────────────┘                   └─────────────────────────────────┘
        ↑                                                    ↑
   iptables перехватывает                              iptables перехватывает
   весь исходящий трафик                               весь входящий трафик
```

**Envoy** (sidecar) делает всё: service discovery, load balancing, retries, timeouts, circuit breaker, mTLS, observability. Приложение думает, что ходит напрямую.
### Архитектура: Data Plane + Control Plane
```plain
┌─────────────────────────────────────────────────────────────┐
│                      CONTROL PLANE                           │
│  ┌─────────┐    ┌─────────┐    ┌─────────────────────────┐  │
│  │ Istiod  │    │ Consul  │    │ Linkerd control plane   │  │
│  │ (xDS)   │    │ server  │    │                         │  │
│  └────┬────┘    └────┬────┘    └─────────────────────────┘  │
│       │              │                                        │
│       └──────────────┴────────────────┐                     │
│              gRPC xDS API               │                     │
└─────────────────────────────────────────┼───────────────────┘
                                          ▼
┌─────────────────────────────────────────────────────────────┐
│                       DATA PLANE                             │
│                                                              │
│  Pod A                    Pod B                    Pod C     │
│  ┌─────────┐             ┌─────────┐             ┌────────┐│
│  │ App     │             │ App     │             │ App    ││
│  │ Envoy   │◄──────────►│ Envoy   │◄──────────►│ Envoy  ││
│  │(sidecar)│   mTLS      │(sidecar)│   mTLS      │(sidecar)││
│  └─────────┘             └─────────┘             └────────┘│
│       ▲                       ▲                       ▲     │
│       └───────────────────────┴───────────────────────┘     │
│                    Service Discovery (xDS)                     │
│              (EDS: endpoints, CDS: clusters,                 │
│               RDS: routes, LDS: listeners)                   │
└─────────────────────────────────────────────────────────────┘
```
#### **Что спросят на собесе?**

- **Зачем Service Mesh, если есть Nginx/HAProxy?**
	- Nginx/HAProxy — **edge proxy** (периметр). Service Mesh — **east-west** (между сервисами). Nginx не знает о Kubernetes pod'ах в real-time, не делает client-side LB per-request, не предоставляет end-to-end mTLS без терминации. Service Mesh решает service-to-service communication в динамической среде.
- **Почему не сделать client-side LB в коде приложения?**
	- Приложение на Go может использовать `grpc.WithDefaultServiceConfig` для client-side LB. Но: (1) нужно реализовывать на каждом языке, (2) нет единого control plane для конфигурации, (3) нет автоматического mTLS, (4) observability размазана. Service Mesh даёт **единый слой для всех языков**.
- **Какой overhead у Service Mesh?**
	- Sidecar: +3-5ms latency, +~100MB RAM на pod, +CPU на TLS handshake. Ambient mode снижает до +1-2ms за счёт eBPF. Для высокочастотных low-latency систем (HFT, игровые серверы) может быть критично — тогда без mesh или с eBPF.

## **Ввод URL в браузере**

1) Ввод URL в браузере
	- браузер парсит scheme, host, port
2) проверка кэшей
	- HTTP-кэш браузера
	- DNS кэш браузера
	- DNS кэш операционной системы
3) резолвинг имени
	- системный резолвер – getaddrinfo (или более старый вариант – gethostbyname)

	- запрос уходит на DNS-сервер из /etc/resolv.conf

	- для HTTP – TCP three-way handshake (syn, syn+ack, ack)

	- браузер отправляет HTTP request, получает HTML
7) рендеринг
	- парсинг HTML -> DOM
	- загрузка ресурсов (css, js, картинки)

##  **HTTP/2**

> **Это бинарный протокол поверх TCP, который решает проблемы производительности HTTP/1.1 за счёт мультиплексирования, сжатия заголовков и управления приоритетами загрузки ресурсов.** НО head-of-line blocking на уровне TCP.

##### **Head-of-line blocking**

**Head-of-line blocking (HOL) на уровне TCP** — это ситуация, когда потеря **одного** TCP-сегмента блокирует доставку **всех последующих данных**, даже если они уже получены.

##### **Почему так происходит?**

TCP гарантирует:
- **надёжность**
- **упорядоченную доставку (in-order delivery)**

в HTTP/3 нет HOL blocking между стримами
##### **HTTP/2 vs HTTP/1.1?**
- наличие мультиплексирования запросов – несколько запросов идут **по одному TCP-соединению параллельно**
- бинарный формат передаваемых данных – для оптимизации
- сжатие заголовков **HPACK** – уменьшает overhead
- **server push** – проталкивать какие-то ресурсы на клиент до того, как он их запросит – для оптимизации загрузки
- приоритизация загрузки ресурсов – для оптимизации
##### **Что важно для go-разработчика?**
- в Go HTTP/2 включается автоматически в `net/http` при использовании TLS

## **HTTP3 (QUIC)**

- быстрая установка соединения (TCP требует рукопожатия, а это время (RTT-round-trip-time) при удаленности клиента и сервера >100мс)
- меньше блокировок HoL
- QUIC использует использует протокол UDP, потому что это позволяет не обновлять огромное количество устройств, которые уже знакомы с UDP, то есть так QUIC было проще развернуть

## **NAT**

> **Network Address Translation** - это механизм в сетях TCP/IP, позволяющий изменять IP адрес в заголовке пакета, проходящего через устройство маршрутизации трафика. 
> 
> **NAT** преобразует приватный IP-адрес в общедоступный и скрывает внутреннюю сеть за одним IP. Это позволяет устройству с частным адресом IPv4 обращаться к ресурсам за пределами его частной сети.

Когда клиент делает запрос:
```
192.168.0.2:54321 → google.com:443
```
NAT преобразует:
```
PUBLIC_IP:40000 → google.com:443
```
И хранит таблицу:
```
192.168.0.2:54321 <-> PUBLIC_IP:40000
```

## **Polling (short, long), SSE, WebSockets**

> Все эти техники — это способы доставлять обновления от сервера к клиенту в условиях, где изначально HTTP — request/response протокол.

### **Кратко**
- Polling — это периодические запросы клиента.  
- Long polling — сервер держит соединение, пока не появятся данные.  
- SSE — это однонаправленный стрим от сервера к клиенту поверх HTTP.  
- WebSocket — полноценное двунаправленное соединение с низким overhead.
По сути это эволюция от неэффективного опроса к постоянному realtime-соединению.
#### **Выбор**
- SSE (уведомления, стримы)
- WebSocket (чаты, игры, realtime collaboration)

### **Polling (short polling)**
> клиент регулярно шлёт HTTP-запросы: “есть ли новые данные?” – самый простой, но не самый эффективный

#### **Как работает**
```
client → request
server → response (есть/нет)
(повторяем каждые N секунд)
```
**Плюсы**
- простота
- работает везде
**Минусы**
- куча лишних запросов
- высокая latency (между апдейтами)
- нагрузка на сервер

### **Long Polling**
> сервер **держит соединение открытым**, пока не появятся данные.
> Это компромисс: уменьшаем количество запросов, но остаёмся в модели request/response.

#### **Как работает**
```
client → request
server → (ждёт)
server → response (как только есть данные)
client → сразу делает новый request
```
**Плюсы**
- меньше лишних запросов
- почти realtime
**Минусы**
- всё ещё HTTP-overhead
- много открытых соединений
- сложнее масштабировать

### **Server-Sent Events (SSE)**

> Сервер пушит события по одному HTTP-соединению. Однонаправленно: **server → client**.
> SSE — это серверный push поверх HTTP, но только в одну сторону.

#### **Как работает**
```
- HTTP response не закрывается
- сервер стримит текстовый поток (`text/event-stream`)
```
**Плюсы**
- проще, чем WebSocket
- автоматический reconnect в браузере
- хорошо для стриминга событий
**Минусы**
- только в одну сторону
- текстовый формат
- ограничения браузеров

### **WebSocket**
> Полноценное **двунаправленное соединение**. После upgrade от HTTP.
> **WebSocket** — это постоянное двунаправленное соединение с минимальным overhead.

#### **Как работает**
```
1. HTTP request с Upgrade
2. переключение на WebSocket
3. дальше — постоянное соединение
```
**Плюсы**
- full-duplex (client ↔ server)
- низкая latency
- меньше overhead
**Минусы**
- сложнее инфраструктура
- сложнее балансировка
- нужно управлять соединениями
