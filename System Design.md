# Шаг 1: Формализация требований

> Прояснить **функциональные требования** (что система должна делать) и **нефункциональные требования** (как система должна делать).
> Этот список важно уточнить у интервьювера как у заказчика (предлагая свои идеи и задавая осознанные вопросы).
> Например, для банковской системы consistency важнее latency, а для ленты соцсети — наоборот.
> Важно понять scope и границы системы, задать приоритеты фичей.
#### Нефункциональные требования (NFR)
список того, что важно для конкретной системы (не все могут требовать внимания в контексте конкретной системы).
- **High availability** – 99.9% времени
- **Scalability** – способность справляться с ростом нагрузки
- **Perfomance** – например, latency p99 <100мс
- **Durability** – должны ли данные теряться
- **Consistency** – strong vs eventual consistency; осознанно выбираем, нужна ли строгая консистентность или допустимо, что система в какой-то момент будет демонстрировать неконсистентность, но в конечном счете данные станут согласованными спустя какое-то время (eventual consistency)
- **Fault tolerance** – переживет ли система отказы компонентов

- **REST API** – стандарт для публичных API
- **RPC (gRPC)** – стандарт для межсервисных взаимодействий
- **MQ** – для асинхронной коммуникации

# Шаг 3: Core flows и компоненты

> Основные потоки данных в системе. Здорово разделить read/write paths, если это необходимо. Проговорить happy/exceptional paths.

- **Read path** – как данные будут читаться (кэши, реплики)
- **Write path** – как данные записываются (очереди, журналирование)
- **Happy path** – основной успешный сценарий
- **Exceptional flows** – обработка ошибок, ретраи, fallbacks

# Шаг 4: Концептуальная схема данных

> Основные сущности и связи между ними.
> Пока без привязки к конкретным технологиям.
> Проговорить stateful и stateless компоненты.

- **Stateful компоненты** – базы данных, кэши с персистентностью, очереди, файловые хранилища
- **Stateless компоненты** – API серверы, workers/processors, load balancers, gateways

# Шаг 5: Реальная схема данных (технологии)

> Фиксируем технологии. Обосновываем выбор и проговариваем trade-offs.
> Failure domains – продумать, что произойдет при отказе каждого из компонентов. Какой blast radius (масштаб)? Как система восстановится?
> Индексы, денормализация.

- **Базы данных**
	- реляционные – PostgreSQL (ACID, сложные запросы, JSON) vs MySQL (надёжность, репликация)
	- колоночные – Cassandra (высокая доступность, масштаб) vs Clickhouse
	- документные – MongoDB (документы, гибкая схема)
	- графовые – Neo4j
- **Кэширование**
	- Memcached (простой key-value, multi-threading)
	- Redis (структуры данных, pub/sub, персистентность)
	- local cache (in-memory)
- **Очереди сообщений**
	- Kafka (высокая пропускная способность, replay)
	- RabbitMQ (гибкая маршрутизация, AMQP)
	- SQS (managed, serverless)
	- Redis streams
- **Полнотекстовый поиск**
	- Elasticsearch (полнотекстовый поиск, аналитика)
	- Solr
	- Meilisearch (простой, typo-tolerant)
- **Object storage** – S3 vs Minio

# Шаг 6: Масштабирование

> Как системе пережить рост в 10x, 100x, 1000x.
> Проговорить/обозначить понимание разницы верт/гориз масштабирования, стратегий шардирования, bottlenecks системы.
> Подумать о data consistency системы после масштабирования.

- **Вертикальное масштабирование** – увеличение мощности одной машины (проще, но ограниченно; Single point of failure)
- **Горизонтальное масштабирование** – добавление серверов (сложнее, но безгранично; Требует stateless дизайна)
- **Партицирование данных** – разбиение данных по ключу (user_id, region, time)
- **Шардинг** – разбиение данных по нескольким базам (на одной или нескольких машинах)
- **Репликация** – копии данных для чтения и отказоустойчивости
- **CQRS** – разделение моделей чтения и записи
- **Консистентное хэширование** – минимизация перераспределения при добавлении/удалении нод
